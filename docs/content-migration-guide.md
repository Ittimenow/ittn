# Инструкция по переносу кейсов со старого сайта

Документ описывает, как переносить кейсы со старого сайта `ittimenow.com` в текущий Astro-проект.

Главный принцип: **контент не переписывать**. Можно менять только структуру, верстку и расположение блоков под формат нового сайта.

## Где лежат кейсы

- Контент кейсов: `src/content/cases/*.md`
- Медиа кейсов: `public/cases/<case-slug>/`
- Страница кейса: `src/pages/cases/[...slug].astro`
- Схема данных: `src/content.config.ts`

Новый кейс добавляется отдельным Markdown-файлом в `src/content/cases`.

## Базовый порядок переноса

1. Открыть старую страницу кейса и сохранить исходный HTML или текст.
2. Извлечь весь текстовый контент без переписывания.
3. Выписать все медиа: изображения, видео, аватары, схемы, скриншоты.
4. Скачать медиа локально в `public/cases/<case-slug>/`.
5. Создать Markdown-файл кейса в `src/content/cases/<case-slug>.md`.
6. Разложить контент по обязательным секциям нового сайта:
   - `Клиент`
   - `Задача`
   - `Результаты`
   - `Процесс`
   - `Команда`
7. Внутри этих секций можно сохранять дополнительные старые подзаголовки, если они нужны по смыслу.
8. Собрать проект и проверить, что страница кейса и список кейсов работают.

## Frontmatter

Минимальный формат:

```md
---
title: 'Название кейса'
client: 'Название клиента'
summary: 'Краткое описание для карточки и meta description.'
industry: 'Отрасль'
year: 2025
stack: ['Технология 1', 'Технология 2']
services: ['Услуга 1', 'Услуга 2']
featured: true
order: 8
cover: '/cases/<case-slug>/cover.png'
---
```

Если в качестве верхнего cover нужно использовать видео:

```md
cover: '/cases/<case-slug>/cover-poster.png'
coverVideo: '/cases/<case-slug>/cover-video.mp4'
```

`cover` в этом случае остается постером для видео.

## Правила по тексту

- Не переписывать формулировки.
- Не улучшать стиль текста без отдельной просьбы.
- Не менять смысл, цифры, названия систем, имена и должности.
- Допустимо менять уровень заголовков, если это нужно для структуры нового сайта.
- Допустимо переносить старый блок внутрь ближайшей обязательной секции.

Пример:

- Старый блок `Предпосылки` можно положить внутрь `## Задача`.
- Старый блок `Технологии` не добавлять в тело кейса, если пользователь просит убрать его из контента. Стек можно оставить в frontmatter.
- Старый блок с перечнем работ оформить как список работ, а не как обычный текст.

## Правила по медиа

- Все медиа переносить локально в `public/cases/<case-slug>/`.
- В Markdown использовать абсолютные пути от `public`:

```md
![Описание](/cases/<case-slug>/image.png)
```

```html
<video class="case-video" autoplay loop muted playsinline>
  <source src="/cases/<case-slug>/video.mp4" type="video/mp4" />
</video>
```

- Если на старом сайте есть несколько видео с похожими именами, проверять, какое именно относится к конкретному блоку.
- Для autoplay-видео без кнопок управления не добавлять `controls`.
- Для видео с пользовательским управлением добавлять `controls` только если это явно нужно.
- Видео вертикального формата не увеличивать до ширины страницы. Обычно это скринкаст мобильной версии, рядом с которым может быть текстовое пояснение.

## Структура обязательных секций

### Клиент

Краткое описание клиента со старого сайта.

Если на старом сайте после клиента стояло hero-видео, не всегда нужно оставлять его в теле секции. Если пользователь просит видео на место cover, использовать `coverVideo`.

### Задача

Сюда переносить:

- старую `Задачу`;
- `Предпосылки`;
- список проблем клиента;
- формулировки целей.

### Результаты

Сохранять пункты результатов как есть.

Если на старом сайте результаты оформлены галочками `✔`, можно оставить их отдельными абзацами или списком. Не добавлять сюда технический стек, если он визуально выглядит лишним.

### Процесс

Сюда переносить основную историю работы:

- аналитику;
- настройку системы;
- слайдеры;
- отдельные функциональные блоки;
- скриншоты интерфейсов;
- видео процесса.

### Команда

Использовать существующий HTML-паттерн:

```html
<div class="case-team">
  <div class="case-team-card">
    <img src="/cases/<case-slug>/avatar-name.jpg" alt="" />
    <h4>Роль</h4>
    <p>Имя Отчество</p>
  </div>
</div>
```

## Паттерны верстки

### Слайдер изображений

Для галерей использовать существующий `case-slider`:

```html
<section class="case-slider" data-case-slider aria-label="Название слайдера">
  <div class="case-slider-viewport">
    <div class="case-slider-track">
      <div class="case-slider-slide">
        <img src="/cases/<case-slug>/image-1.png" alt="Описание" />
      </div>
    </div>
  </div>
  <div class="case-slider-controls">
    <button class="case-slider-button case-slider-button-prev" type="button" aria-label="Предыдущий слайд">&larr;</button>
    <div class="case-slider-dots" aria-label="Навигация по слайдам">
      <button class="case-slider-dot" type="button" aria-label="Слайд 1"></button>
    </div>
    <button class="case-slider-button case-slider-button-next" type="button" aria-label="Следующий слайд">&rarr;</button>
  </div>
</section>
```

Количество dots должно совпадать с количеством слайдов.

### Блок списка работ

Если старый сайт показывает заголовок слева, а список работ справа, использовать отдельный HTML-блок:

```html
<section class="case-work-list">
  <div class="case-work-list-title">
    <h3>Первичная настройка системы</h3>
  </div>
  <ol>
    <li>Проверка базы 1С</li>
    <li>Настройка прав доступа сотрудникам</li>
  </ol>
</section>
```

Этот блок нужен, когда текст является именно списком работ, а не обычным описанием.

### Блок с боковым текстом и изображениями

Если на старом сайте текст расположен сбоку от изображения, не превращать это в обычную последовательность `текст -> картинка -> текст`.

Использовать HTML-секцию с рядами:

```html
<section class="case-waybills">
  <div class="case-waybill-row">
    <div class="case-waybill-copy">
      <p>Текст слева.</p>
    </div>
    <div class="case-waybill-media">
      <img src="/cases/<case-slug>/image.png" alt="Описание" />
    </div>
  </div>

  <video class="case-waybill-video" autoplay loop muted playsinline>
    <source src="/cases/<case-slug>/video.mp4" type="video/mp4" />
  </video>

  <div class="case-waybill-row is-reversed">
    <div class="case-waybill-media">
      <img src="/cases/<case-slug>/image-2.png" alt="Описание" />
    </div>
    <div class="case-waybill-copy">
      <p>Текст справа.</p>
    </div>
  </div>
</section>
```

Использовать этот подход для блоков вроде `Путевые листы`, где на старом сайте композиция была:

- текст сбоку;
- большая схема;
- видео;
- вторая схема с подписью сбоку.

### Блок интерфейсных фишек

Если старый сайт показывает несколько пар `скрин интерфейса + пояснение`, использовать:

```html
<section class="case-interface-features">
  <div class="case-interface-feature">
    <div class="case-interface-media case-interface-media-small">
      <img src="/cases/<case-slug>/notice.png" alt="Описание" />
    </div>
    <div class="case-interface-copy">
      <h4>Заголовок</h4>
    </div>
  </div>

  <div class="case-interface-feature is-reversed">
    <div class="case-interface-copy">
      <p>Текст пояснения.</p>
    </div>
    <div class="case-interface-media">
      <img src="/cases/<case-slug>/screen.png" alt="Описание" />
    </div>
  </div>
</section>
```

Использовать `case-interface-media-small` для маленьких уведомлений, которые не должны растягиваться на всю ширину.

## Когда добавлять стили

Если нужного блока нет в дизайн-системе, стили добавлять в `src/pages/cases/[...slug].astro` рядом с другими case-specific стилями.

Не создавать новый глобальный компонент, если блок нужен только одному кейсу.

Новый стиль должен:

- использовать текущие переменные и визуальный язык сайта;
- не копировать старую Divi-стилистику один в один;
- сохранять композицию старого сайта;
- быть адаптивным;
- не ломать остальные кейсы.

## Проверка после переноса

Обязательные команды:

```bash
npm run build
```

Проверить локальные пути к медиа:

```bash
node --input-type=module -e "import { readFileSync, existsSync } from 'node:fs'; const md = readFileSync('src/content/cases/<case-slug>.md','utf8'); const refs = [...md.matchAll(/(?:src=\"|\\]\\()([^\\\")]+)(?:\"|\\))/g)].map(m=>m[1]).filter(p=>p.startsWith('/cases/')); const missing = refs.filter(p=>!existsSync('public'+p)); console.log(JSON.stringify({refs: refs.length, uniqueRefs: new Set(refs).size, missing}, null, 2));"
```

Если сборка показывает старый HTML после изменения Markdown, очистить Astro-кэш:

```bash
rm -rf .astro
npm run build
```
